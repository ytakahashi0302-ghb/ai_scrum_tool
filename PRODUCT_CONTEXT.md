# PRODUCT_CONTEXT.md — MicroScrum AI プロダクトコンテキスト

> このドキュメントはAIへの引き継ぎ専用です。
> チャットを変える際は必ず冒頭にペーストしてください。
> スプリントを重ねるごとに育てていくドキュメントです。

---

## このプロダクトが解決したいこと

ソロ開発者がAIと一緒に開発を進める際、以下の問題がある：

- AIツール（Devin・Claude Codeなど）は「全自動」か「都度指示」の二択で、人間が主体的に関わり続ける仕組みがない
- 開発が進むにつれてAIへの指示が毎回ゼロから始まり、文脈が蓄積されない
- 人間とAIの「共通言語」がないため、AIが何をしているか分かりにくく不安が残る

**MicroScrum AIはスクラムを人間とAIの共通プロトコルとして使うことで、これらを解決する。**

---

## 誰のためのツールか

- **主対象**：ソロ開発者（個人開発・副業・スタートアップ初期）
- **前提スキル**：スクラムの基本概念を知っている、またはこのツールで学べる人
- **前提環境**：ローカルで動かしたい・プライバシーを重視したい人

チーム開発向けツール（Jira・Linear）でも、全自動AIツール（MetaGPT・Devin）でもない。
**「人間がオーナーで、AIチームと一緒に育てる」ポジションを狙っている。**

---

## プロダクトの核心的な思想

### 1. スクラムは共通言語である

スクラムを採用している理由は「アジャイルだから」ではない。
**人間とAIがお互いの行動を予測・理解できる共通プロトコルとして機能するから。**

- 人間側：スクラムを知っていれば、AIが次に何をするか分かる → 安心して任せられる
- AI側：状態遷移が明確なので、曖昧な指示でも動ける
- お互い：レトロで振り返れる → スプリントごとに改善できる

### 2. AIチームは育てるものである

1回使い捨てのAI（MetaGPT的な一発生成）ではなく、
**スプリントを重ねるごとに自分の開発スタイルに最適化されていくAIチーム**を目指す。

各エージェントがレトロで振り返り、その内容をSQLiteに蓄積し、
次スプリントのプランニングに自動反映することで実現する。

### 3. 人間はアイディアに集中する

アイディアメモ → User Story → タスク分解 → 実行 → レトロ という流れで、
**人間が関与するのはアイディアと最終判断だけ**にする。
中間プロセスはすべてAIが担う。

### 4. 透明性がなければ任せられない

AIが何をしているかブラックボックスでは、人間は不安で任せられない。
スクラムのカンバン・スプリントゴール・操作スコープ宣言によって、
**AIの行動を常に可視化する**。

## スクラムイベントの設計判断

通常スクラムの5イベントをソロ×AI向けに**3イベントに再設計**している。

```
通常スクラム → MicroScrum AI
─────────────────────────────────────────
スプリントプランニング   →  プランニング（壁打ちでリファインメントを統合）
バックログリファインメント → （プランニングに統合）
デイリースクラム        → （カンバンで代替・省略）
スプリントレビュー      → レビュー＋レトロ（セット）
レトロスペクティブ      → （レビューに統合）
```

**なぜ統合したか：**
- リファインメントは壁打ちエージェントとの会話で自然に実現できるため独立不要
- デイリーはソロ開発では不要（カンバンで常に進捗が見える）
- レビューとレトロはスプリント完了時にセットでやるのが自然なリズム

## 各エージェントの設計判断

### スクラムマスターの設計方針

**SMは「意思決定の境界線」にだけ登場する独立エージェント**として設計する。

人間チームではSMとDevの兼任はできないが、AIはコンテキストの注入量を変えるだけで
「別人格」に切り替えられる。これはAIならではのアプローチ。

#### なぜSMにフラットな視点が必要か

プロダクトコンテキストをフルに注入されたエージェントは「その文脈で最適」に動く。
これは実行においては正しいが、**「そもそもこの方向性でいいか」を疑えない**という問題がある。
コンテキストに染まっていないSMだからこそ、エコーチェンバーを壊せる。

#### SMのコンテキスト注入設計

```
実行系エージェント（壁打ち・タスク分解・実行・レビュー）
├── PRODUCT_CONTEXT.md（フル）
├── ARCHITECTURE.md
└── 直近のタスク情報

SMエージェント
├── プロダクト名と目的（1〜2行のみ）
├── 今スプリントのゴール
└── チェック観点のリスト
    ※詳細設計・過去の判断経緯は渡さない
```

#### SMが登場するタイミング

```
プランニング前半（壁打ち・タスク分解）
└── 実行系エージェントが担当

プランニング後半（人間が承認する直前）  ← SMが登場
└── 完成の定義は明確か
    優先順位の根拠はあるか
    スコープは適切か

スプリント実行中
└── SMなし（実行系エージェントが担当・カンバンで可視化）

レトロ  ← SMが再登場
└── スプリントの方向性は正しかったか
    改善できた部分はどこか
    次スプリントへの提言
```

人間チームで言えば「外部コーチが節目だけ来る」感覚に近いが、
コストゼロ・切り替え即時でできるのはAIならでは。

### 開発チームを複数エージェントにする価値がある場合

複数エージェントにする意味があるのは「品質チェック機構」として機能させるとき：
- レビュアー役（実装したコードを別エージェントが検査）
- 専門特化（セキュリティ・テスト・ドキュメント）

単純に並列実行するためではない。

### バックログリファインメントをプランニングに統合した理由

人間チームでリファインメントが必要な理由（背景知識の共有・認識合わせ）は
AIにはそのまま当てはまらない。AIはコンテキストを共有済みで動けるため、
プランニング前の自動フェーズとして吸収できる。

## ベロシティの単位はストーリーポイントではなく「時間と予算」

AIにとっての「キャパシティ」はトークン＝お金であり、時間は人間のリズム。
この2軸でスプリントを管理する：

```
予算（¥）→ コストの上限。AIへの依存度を制御する。
時間（h）→ 人間のリズム。区切りを作る。1時間後に成果を見たい。
```

どちらかに達した時点でスプリント完了とする。
レトロでも「時間効率」と「コスト効率」の両方を振り返り指標とする。

### コストボックスの目安は「育てる」設計

比率で割り当てる人間スクラムのタイムボックスとは発想が異なる。
イベント（SMが考える）のトークンコストは小さく、実行（AIが動く）のコストが大半を占めるため、
**イベントは固定コスト・実行は変動コスト**として設計する。

```
イベント（固定）
├── プランニング：上限¥XX固定
├── レビュー：上限¥XX固定
└── レトロ：上限¥XX固定

スプリント実行（変動）
└── 予算残額をすべて使う
    例）¥200スプリントなら残額が実行予算
```

**初期値は計測スプリントで決める。**
最初の1スプリントは予算上限を設けず実績を計測し、
「このタスク規模で¥XXかかった」という自分のベロシティ基準値を把握する。
以降はレトロで実績を振り返りながら予算目安を更新していく。

## 承認フローの設計判断

「全自動（毎回確認なし）」と「毎回確認」の二択ではなく、
**スプリント開始時の一括承認 × 操作の危険度による自動分類**を採用する。

理由：Antigravityのように毎回確認するツールは使い勝手が悪い。
Claude Code desktopのような快適さを、スクラムのスプリントという単位で実現する。

- スプリント開始時に操作スコープを宣言・人間が一度だけ承認
- スプリント中は宣言範囲内を全自動実行
- 危険度HIGH（削除・範囲外アクセス）のみ必ず止まる

## 仕様駆動開発との差異

仕様駆動開発（Spec-Driven Development）と構造が近く見えるが、前提が異なる。

```
仕様駆動開発
├── 仕様を先に完全に定義する
├── AIはその仕様に従って実装する
└── 仕様 → 実装（一方向）

MicroScrum AI
├── PRODUCT_CONTEXT・RULE・ARCHITECTUREが「生きた仕様」
├── スプリントごとに仕様自体が育っていく
└── 仕様 ⇄ 実装 ⇄ レトロ（循環）
```

**決定的な違いは「仕様は常に不完全である」という前提に立っているかどうか。**

仕様駆動開発は仕様の完成度がそのままプロダクトの完成度になる。
MicroScrum AIはSMのフラットな視点・レトロの振り返りを通じて
**仕様の間違いや漏れをスプリントの中で発見することに価値がある**と考える。

スクラムをあえて採用している理由の一つがここにあり、
不完全な仕様でも動き始められる設計になっている。

## やらないことリスト

- **チーム向け機能**：複数人でのリアルタイムコラボレーション
- **クラウド同期**：データは完全ローカル（SQLite）に留める
- **全自動化**：人間の関与ポイント（プランニング・レビュー）は省略しない
- **汎用タスク管理**：スクラムのフレームを外れた使い方はサポートしない
- **SM常駐化**：意思決定の境界線（プランニング後半・レトロ）にのみ登場させる。実行中は不要

## コンテキスト引き継ぎの仕組みについて

このツールの開発自体が「チャットのコンテキスト管理」問題を抱えている。
同じチャットを続けると精度は上がるがレート制限に引っかかる。
チャットを変えると文脈が失われて精度が下がる。

これはツールとしても解決すべき課題であり、以下の2層構造で対応する予定：

```
Layer 1：プロダクトコンテキスト（このファイル・永続）
└── なぜ作るか・設計判断の理由・やらないこと

Layer 2：スプリントコンテキスト（handoff.md・毎回更新）
└── 今どこまで進んでいるか・次のAIへの指示
```

このファイル（Layer 1）はすべてのチャットの冒頭に必ずペーストする。
handoff.md（Layer 2）はスプリント単位で更新し、続きから始める際に使う。

## 技術スタック（変更時は必ず理由も記録する）

| レイヤー | 技術 | 選定理由 |
|----------|------|----------|
| フロントエンド | React 19, TypeScript, Tailwind CSS v4 | - |
| バックエンド | Tauri v2 (Rust) | ローカル動作・セキュリティ・軽量 |
| データベース | SQLite（ローカル） | プライバシー重視・オフライン動作 |
| AI | Anthropic API, Gemini API | マルチモデル対応で柔軟性確保 |

**Tauriを選んだ理由の補足：**
フロントエンドに直接のシェル実行権限を与えず、Rust側でバリデーションを挟む設計が
AIエージェントの誤操作リスクを防ぐ上で適している。

## 開発スタイル

このプロダクト自体もAIエージェントで開発している：

- **Gemini 3.1 Pro（ブラウザ）**：エージェントへの指示プロンプトを設計・生成
- **Antigravity × Gemini 3.1 Pro High**：コード生成・ファイル操作を実行

「AIに何をさせるかを考えるAI」と「実際に手を動かすAI」を分けるこの構造は、
このツールが目指す人間とAIの役割分担そのものでもある。
