# Epic 22: Agent Orchestratorの統合と自律的タスク生成 実装計画

ユーザーの曖昧な要望をAIが解釈し、自律的にストーリーおよびサブタスクに分解してカンバン（DB）に自動登録するフローを確立するための実装ロードマップです。

## User Review Required

> [!IMPORTANT]
> **アーキテクチャの方向性について**
> 現在 `chat_with_team_leader` APIは文字列の `reply` のみを返却するシンプルなチャットボットとして動作しています。
> 今回の実装では、会話を通じて「システムへの介入（DBへのデータ書き込み）」を伴う Orchestrator レイヤーへ進化させます。
> 
> 具体的なアプローチとして、以下の**2段階パイプライン（Agent Orchestrator）**をバックエンド（Rust）に実装することを提案します。
> これによりフロントエンドへの負担を減らし、Rust側で整合性を持ったDB更新を行います。
> 1. AIが意図を汲み「これはタスク化すべき内容か」を判定し、構造化データ（Story 概要 + Task群）を生成。
> 2. そのデータをもとにRust側でDBに `stories` テーブルと `tasks` テーブルへのINSERT処理を一括で実行。
> 3. AIチャットとして完了報告（「XXXのストーリーと3つのタスクをバックログに登録しました」）の文字列と、フロントエンドへのリロード要求シグナルを返却。

> [!WARNING]
> この作業において、フロントエンド側（特に `TeamLeaderSidebar.tsx` やカンバンのフェッチ処理）との連携が必要になります。TauriのEventシステム（`app.emit`）を利用して更新シグナルを飛ばすか、APIのレスポンスにフラグを持たせてReactステートの再取得（refetch）をトリガーする方針とします。

## Proposed Changes

---

### Phase 1: AI Agent Orchestratorのバックエンド設計 (Rust)

AIのレスポンス構造を拡張し、単なるテキストトークだけでなく「実行アクション」を含められるようにします。

#### [MODIFY] `src-tauri/src/ai.rs`
- 新たに `ChatOrchestratorResponse` 構造体（仮）を定義。
- `reply: String` に加え、`action_required: Option<ActionPayload>` のような形でタスクの起票を伴うデータの型を定義。
- または、既存の関数の内部実装で**Rigパイプライン**を組み上げ、「①意思決定・ストーリー生成」->「②タスク分解」を連続でLLMに実行させる処理（Agent Workflow）を追加。

---

### Phase 2: DB操作とのトランザクション統合 (Rust)

生成された一連のタスクをアトミックにDBに保存する機能を準備します。

#### [MODIFY] `src-tauri/src/db.rs`
- 既存のDB挿入関数を活用し、Orchestratorから直接「Storyの作成 → Task群の作成」を紐づけて行うヘルパー関数を追加。

---

### Phase 3: フロントエンドとAI連携強化 (React)

サイドバーからの指示に対するレスポンス処理と、カンバン更新を同期させます。

#### [MODIFY] `src\components\ai\TeamLeaderSidebar.tsx`
- サイドバーでのチャット送信時、AIからのレスポンスを解析。
- タスク起票アクションが含まれていた場合、チャット履歴に表示すると共に、ワークスペースコンテキストを通じたカンバンの再フェッチ関数を実行（イベント通知のハンドリング）。

#### [MODIFY] Kanban Context or Event Listener
- Tauriイベントを購読、またはサイドバー起因の更新トリガーを検知してデータソースをリフレッシュする処理の実装。

> Rigの設定で、使用モデル（Gemini 3.1 Proなど）のTool Calling(関数呼び出し)機能を本格的に利用するか、あるいはプロンプトエンジニアリングでのJSON出力強要とパースで擬似的にパイプラインを組む形にするかの選定が必要です。
> 今回は既存機能の延長であるため、まず **JSON構造化出力を用いたパイプライン型（プロンプトベース）** で堅牢に組む形を想定していますが、よろしいでしょうか？
