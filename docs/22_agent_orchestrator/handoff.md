# Epic 22: Agent Orchestratorの統合と自律的タスク生成 (Handoff)

## 1. 実装完了サマリー
本Epicでは、ユーザーの曖昧な要求・アイデアをAIが自律的に適切なバックログタスクへ変換し、UIに即座に反映させる「エージェント・オーケストレーター基盤」の構築を完了しました。

- **`rig-core` Tool Callingを用いた自律的なDB登録ロジック**
  AIエージェントに `CreateStoryAndTasksTool` を付与し、既存ストーリーへのタスク追加や新規機能のストーリー群起票を解釈させ、Rustバックエンド上で同期的に一括でSQLiteデータベースに永続化するパイプラインを確立しました。
- **Tauri Eventを用いたバックエンドとフロントエンドの疎結合同期**
  データ更新（INSERT）が成功した際に、RustからTauriの `emit` を用いて `kanban-updated` イベントを発火。React側（`ScrumContext.tsx`）でそれを `listen` することで、手動リロードなしでフワッとカンバンボードが最新化されるモダンなUXを実現しました。

## 2. 【警告】技術的な重要ポイント (罠の回避録)
次期開発者が同じ罠に陥らないよう、以下の点に強く注意してください。

> [!CAUTION]
> **SQLiteの `CHECK` 制約の厳密さ**
> `stories` および `tasks` テーブルには、`status` カラムなどに `status IN ('Backlog', 'Ready', 'In Progress', 'Done')` といった非常に厳密な `CHECK` 制約がマイグレーション時（`3_add_projects.sql` 等）に定義されています。
> プロンプトを通じてAIの出力スキーマにステータス文字列を委ねると、微細な揺れ（例: `Todo` と `To Do` の違い）でもSQL実行時に即座にSQLite制約エラーとなります。**必ずRustのクエリ実行レイヤー（`db.rs`）で正しいステータス初期値をハードコードしてバインド（`bind`）する**ように設計してください。

> [!WARNING]
> **Tauri開発におけるRustプロセスの再起動忘れ**
> フロントエンド（React / TypeScript）側のコードは保存時に自動でホットリロード（Vite）が効きますが、バックエンド（Rust）側の実装（とくに `ai_tools.rs` や `db.rs` の修正）を行った場合は、**必ずターミナルで `Ctrl + C` を押して現在のTauriプロセスを完全に停止させ、`npm run tauri dev` で再コンパイル・再起動**してください。これを怠ると、ロジックやログ出力の変更がいつまでも反映されず、デバッグ迷子になります。

## 3. 将来の課題 (統合BACKLOG参照)
本Epicでは「ストーリーとタスクの追加機能」にフォーカスしましたが、エージェントが高度化するにつれて以下の対応が必要となります。

- **既存タスクの「更新」や「削除」への対応**
  現在はタスク生成・追加ツールのみですが、今後は `UpdateTaskTool` や `DeleteStoryTool` を追加し、ステータスの自律的変更や誤ったタスクの削除機能を持たせる必要があります。
- **エラー時のロールバック処理とUI通知**
  万が一DB保存処理中にSQLiteがエラーを吐いた場合のエラーテキストは現在ターミナルとAIへ送られていますが、フロントエンド側のUI（トースト通知など）に「処理が中断されました」とユーザーへ視覚的にフィードバックする仕組みの実装が必要です。
