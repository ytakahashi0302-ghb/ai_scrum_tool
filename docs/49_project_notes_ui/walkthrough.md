# EPIC49 実装ウォークスルー

## 実装ハイライト

- `src/components/ai/PoAssistantSidebar.tsx` に `チャット / ふせん` タブを追加し、チャット入力中でもタブ切替で内容が失われない状態管理を導入した。
- `src/components/ai/NotesPanel.tsx` を新規作成し、PO が思考を止めずに書ける軽量なふせん UI を実装した。
- `src/components/kanban/RetrospectiveView.tsx` と連携し、ふせんから Keep / Problem / Try へ直接転記できる導線を完成させた。

## リファインの軌跡

### 1. 「ノート」から「ふせん」へ概念を変更

- 初期計画では「ノート」UIとして、見出しと本文を分けた比較的しっかりした入力フォームを想定していた。
- PO レビューを通じて、短時間スプリント中の気づきは「整理して書く」よりも「思いついた瞬間に残す」ことが重要だと再定義された。
- そのため、UI 概念を「ノート」から「ふせん」へ切り替え、タイトル入力は廃止し、本文のみを素早く追加できる単一メモ形式へ寄せた。
- DB 上は `project_notes` をそのまま利用しつつ、UI 上の表現は全面的に「ふせん」で統一した。

### 2. 削除 UX の改善

- 当初は削除時にネイティブの `window.confirm()` を使っていたが、実操作では削除確認の前後関係が分かりづらく、安心して操作しにくかった。
- そこでネイティブダイアログをやめ、ふせんカード内に「このふせんを削除しますか？」と `削除する / キャンセル` を並べるインライン確認 UI に変更した。
- この変更により、ユーザーの視線移動を最小限に抑えつつ、削除の意図確認をその場で完結できるようになった。

### 3. レトロ転記後の状態管理を整理

- レトロ転記後にふせんが一覧へ残り続けると、「まだ未処理なのか」「転記済みなのか」が曖昧になりやすかった。
- そこで、レトロへ追加したふせんは成功後に `project_notes` 一覧から削除し、サイドバー上では自然に姿を消すフローへ変更した。
- あわせて `vicara:retro-items-updated` イベントを追加し、レトロ画面が開いたままでも `RetrospectiveView` 側が即時 `fetchItems()` を実行するようにした。
- これにより、「転記したのに見えないので不安」という待ち時間をなくし、サイドバーとレトロ画面の状態がその場で揃うようになった。

### 4. 転記時の source 出し分け

- ふせんの作成元は `project_notes.source` で保持している。
- レトロ転記時には、この作成元に応じて `retro_items.source` を出し分けるようにした。
  - 手入力ふせん: `user`
  - PO アシスタント由来ふせん: `po`
- これにより、レトロ画面上でも「PO アシスタントの提案」なのか「PO 本人の手入力」なのかが表示バッジから自然に判断できる。

## UX 方針

- 今回の実装では一貫して、「思考を止めない」「操作をその場で完結させる」「結果をすぐ見せる」を最優先した。
- 具体的には、Quick Switch の一時非表示、単一メモ入力、インライン削除確認、インライン転記カテゴリ選択、即時反映イベントがその方針に対応している。
- 今後の AI 補助機能でも、モーダルやページ遷移でユーザーの流れを分断するより、既存画面の中で自然に処理が完了する構成を優先するのが望ましい。
