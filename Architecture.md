## コアコンセプト

### スクラムを人間とAIの共通プロトコルとして使う

スクラムは「人間チームのためのフレームワーク」として生まれましたが、
このツールでは**人間とAIが協働するための共通言語**として再解釈します。

- 人間は「何を作るか」に集中する（プロダクトオーナー的役割）
- AIは「どう作るか・どう進めるか」を担う（開発チーム的役割）
- スクラムのプロセスが、その協働の透明性を担保する

---

## スクラムイベントの設計方針

通常スクラムの5イベントをソロ×AI向けに3つに再設計します。

```
通常スクラム（5イベント）         MicroScrum AI（3イベント）
─────────────────────────────────────────────────────
スプリントプランニング        →   プランニング
バックログリファインメント    →   （プランニングに統合）
デイリースクラム             →   （カンバンで代替・省略）
スプリントレビュー           →   レビュー＋レトロ（セット）
レトロスペクティブ           →   （レビューに統合）
```

**削減の理由：**
- バックログリファインメントは壁打ちエージェントとの会話で自然に実現できる
- デイリースクラムはソロ開発では不要（カンバンで進捗が常に見える）
- レビューとレトロは完了時にまとめて実施するのが自然なリズム

## エージェント設計方針

### ロール設計の考え方

スクラムの「スクラムマスター」「開発チーム」を別エージェントとして独立させるのではなく、
**各イベントの進行役として機能を溶け込ませる**設計とします。

独立したエージェントを増やすと管理コストと会話の複雑さが増す割に、
ソロ開発では恩恵が薄いためです。

### 実装予定のエージェント

| エージェント | タイミング | 役割 |
|---|---|---|
| 壁打ちエージェント | プランニング時 | アイディア → User Story への対話的整理 |
| タスク分解エージェント | プランニング時 | User Story → Tasks の自動生成・粒度チェック |
| 実行エージェント | スプリント実行時 | タスクの自律実行（ファイル操作・コード生成など） |
| レビューエージェント | レビュー時 | 成果物の品質チェック・受け入れ基準の確認 |
| レトロエージェント | レトロ時 | 各担当の振り返り生成・改善提案 |

### スクラムマスター機能の扱い

SMの機能（プロセスの番人・障害の交通整理・次に何が必要かの整理）は、
各イベントの冒頭処理として組み込みます。

```
プランニング開始時
→ 前回レトロの改善提案を参照し、今スプリントの進め方を調整

スプリント実行中
→ タスクのブロッカーを検出したら人間に通知

レトロ時
→ スプリント全体を俯瞰して障害パターンを整理
```

## レトロスペクティブとDB設計

### 各エージェントが振り返る内容

```
タスク分解エージェント
├── 粒度が大きすぎたタスクはなかったか
├── 見積もりとのズレはどうだったか
└── 依存関係の見落としはなかったか

実行エージェント
├── 詰まったポイントはどこか
├── 想定外の作業が発生したか
└── 次回同じタスクならどう改善するか
```

### SQLiteテーブル設計（予定）

```sql
CREATE TABLE agent_retrospectives (
  id            INTEGER PRIMARY KEY,
  sprint_id     INTEGER NOT NULL,
  agent_role    TEXT NOT NULL,      -- 'task_decomposer' | 'executor' | 'reviewer'
  good          TEXT,               -- うまくいったこと
  bad           TEXT,               -- うまくいかなかったこと
  next_action   TEXT,               -- 次スプリントへの改善アクション
  created_at    DATETIME DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (sprint_id) REFERENCES sprints(id)
);
```

### 次スプリントへの自動反映

プランニング開始時に直近Nスプリント分の`next_action`を取得し、
タスク分解・実行の各エージェントのコンテキストに自動注入します。

```
「前回レトロより：
  - タスク分解エージェント：認証まわりのタスクは細かく分解すること
  - 実行エージェント：外部APIを使うタスクはモックを先に作ること」
```

これにより、スプリントを重ねるほど各エージェントの精度が上がり、
**あなたの開発スタイルに合ったAIチームへと育っていきます。**

## Tauri × AIエージェントのアーキテクチャ

AIエージェントがタスクを自律実行する際の処理境界を明確にします。

```
フロントエンド（React）
│  カンバン表示・進捗確認・人間の操作
│
│  invoke("execute_task", { task_id, instructions })
│
Rustバックエンド（Tauri）         ← 実際の処理はここで行う
├── AI API呼び出し（reqwest）
├── ファイル操作（std::fs）        ← バリデーション後に実行
├── コマンド実行（std::process）   ← 許可リスト制限あり
└── SQLite読み書き（sqlx）
```

**設計方針：フロントエンドに直接のシェル実行権限は与えない。**
AIの指示をRust側で受け取り、バリデーションを挟んでから実行することで、
AIの誤操作によるファイル破壊などのリスクを防ぎます。

## 実行エージェントの承認フロー設計

### 基本方針：毎回確認しない、でも危険な操作は止まる

「全自動」と「毎回確認」の二択ではなく、**スプリント開始時の一括承認 × 危険度による自動分類**を組み合わせます。

毎回確認が必要なツール（Antigravityなど）は使い勝手が悪い一方、
Claude Code desktopのように信頼の文脈が蓄積されたツールは快適に使えます。
この快適さを、スクラムの設計で実現します。

### スプリント開始時の操作スコープ宣言

プランニング完了時にAIが今スプリントで行う操作範囲を提示し、人間が一度だけ承認します。

```
「今スプリントでは以下の操作を行います

  ✅ src/components/ 以下のファイル編集
  ✅ src/hooks/ 以下のファイル新規作成
  ✅ package.json への依存パッケージ追加
  ❌ データベーススキーマの変更は行いません
  ❌ src/ 外のファイルには触りません

  この範囲で進めてよいですか？」
```

**承認後はスプリント中、宣言範囲内を全自動で実行。**
スプリントという単位が、承認スコープの単位を兼ねます。

### 操作の危険度分類

宣言範囲内であっても、操作の危険度に応じて挙動を変えます。

```
危険度 LOW  → 自動実行（ログのみ記録）
  ├── ファイル読み取り
  ├── 宣言済みディレクトリへの新規ファイル作成
  └── プロジェクト内への書き込み

危険度 MID  → 自動実行 ＋ カンバンに通知バッジ表示
  ├── 既存ファイルの編集
  └── 依存パッケージの追加

危険度 HIGH → 承認ゲート（必ず止まる）
  ├── ファイル・ディレクトリの削除
  ├── 宣言範囲外へのアクセス
  └── 外部サービスへの書き込み系API呼び出し
```

### 操作ログとロールバック

実行エージェントの全操作はSQLiteに記録し、スプリント単位でのロールバックを可能にします。

```sql
CREATE TABLE agent_operations (
  id            INTEGER PRIMARY KEY,
  sprint_id     INTEGER NOT NULL,
  task_id       INTEGER NOT NULL,
  operation     TEXT NOT NULL,    -- 'file_write' | 'file_delete' | 'command' など
  target        TEXT NOT NULL,    -- 対象ファイルパス・コマンド
  risk_level    TEXT NOT NULL,    -- 'low' | 'mid' | 'high'
  approved_by   TEXT NOT NULL,    -- 'auto' | 'user'
  snapshot      TEXT,             -- 変更前の内容（ロールバック用）
  created_at    DATETIME DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (sprint_id) REFERENCES sprints(id),
  FOREIGN KEY (task_id) REFERENCES tasks(id)
);
```
